# Tempo

대규모 분산 시스템을 위한 **오픈소스 분산 트래킹(Distributed Tracing) 백엔드**를 의미한다.

<summary>[26.04.09] TraceQL</summary>
<br>

**TraceQL**은 Grafana Tempo에서 사용되는 **분산 트레이싱 전용 쿼리 언어**입니다. Prometheus의 PromQL이나 Loki의 LogQL에서 영감을 받아 만들어졌으며, 수많은 트레이스 데이터 중에서 내가 원하는 특정 조건의 데이터를 빠르고 정확하게 찾아내기 위해 설계되었습니다.

기존의 트레이싱 검색이 단순히 "에러가 발생한 서비스 찾기" 수준이었다면, TraceQL은 "A 서비스의 응답 시간이 1초 이상이면서, B 서비스로 전달된 요청"과 같은 **구조적이고 복잡한 관계**를 검색할 수 있게 해줍니다.

---

### 1. 주요 특징

- **구조적 쿼리:** 서비스 간의 부모-자식 관계, 호출 깊이 등을 조건으로 걸 수 있습니다.
- **유연한 필터링:** HTTP 상태 코드, 리전(Region), 사용자 ID 등 모든 스팬(Span) 속성을 검색 조건으로 활용합니다.
- **집계 기능:** 특정 조건에 맞는 트레이스들의 통계를 내거나 시각화하는 데 유용합니다.
- **익숙한 문법:** PromQL이나 LogQL을 써봤다면 쉽게 배울 수 있는 직관적인 연산자를 사용합니다.

---

### 2. TraceQL 문법 예시

TraceQL은 중괄호 `{}` 안에 조건을 넣는 방식으로 작성합니다.

- **특정 서비스의 에러 검색**

  > 이름이 'auth-service'인 서비스에서 발생한 에러 트레이스 검색

  ```traceql
  { .service.name = "auth-service" && .status = error }
  ```

- **지연 시간(Latency) 조건 검색**

  > 전체 실행 시간이 2초보다 긴 트레이스 검색

  ```traceql
  { .duration > 2s }
  ```

- **연쇄 조건 (부모-자식 관계)**

  > 'gateway' 서비스를 거쳐 'database' 서비스에서 SELECT 쿼리를 실행한 트레이스 검색

  ```traceql
  { .service.name = "gateway" } >> { .service.name = "database" && .db.statement =~ "SELECT.*" }
  ```

---

### 3. TraceQL이 해결하는 문제

1.  **성능 병목 지점 파악:** "전체 경로는 5초가 넘는데, 특정 DB 호출 구간은 100ms 미만인 경우"를 찾아내어 진짜 느린 구간을 필터링할 수 있습니다.
2.  **특정 사용자 경험 추적:** `customer_id` 같은 커스텀 속성을 심어두었다면, "VIP 고객이 겪은 에러"만 골라낼 수 있습니다.
3.  **대량 데이터 탐색:** 인덱스가 없는 Tempo의 특성상 검색이 어려울 수 있지만, TraceQL은 데이터를 효율적으로 스캔하여 수 테라바이트의 데이터 속에서도 원하는 패턴을 찾습니다.

---

### 4. 핵심 구성 요소

| 요소                     | 설명                      | 예시                                 |
| :----------------------- | :------------------------ | :----------------------------------- |
| **Spans**                | 트레이스의 개별 작업 단위 | `{ .http.method = "GET" }`           |
| **Attributes**           | 스팬에 담긴 세부 정보     | `.service.name`, `.http.status_code` |
| **Duration**             | 작업에 소요된 시간        | `.duration > 500ms`                  |
| **Structural Operators** | 스팬 간의 관계 정의       | `>>` (자손), `>` (직계 자식)         |

## 1. 개요

마이크로서비스 아키텍처(MSA)에서는 하나의 요청이 여러 개의 서버를 거쳐 처리됩니다. 이때 "어느 구간에서 병목이 생겼는지" 혹은 "어디서 에러가 났는지" 추적하는 것이 매우 힘든데, Tempo는 이 과정을 추적하는 **추적 데이터**를 저장하는 고성능 **저장소**입니다.

- **비용 효율성:** 고가의 SSD나 메모리 대신 S3, GCS, Azure Blob Storage 같은 **객체 스토리지**에 데이터를 저장하므로 대량의 데이터를 저렴하게 보관할 수 있습니다.
- **통합성:** Grafana, Prometheus, Loki와 완벽하게 연동되어 메트릭(Metric), 로그(Log), 트레이스(Trace)를 하나의 화면에서 볼 수 있게 해줍니다.
- **의존성 제거:** Elasticsearch나 Cassandra 같은 복잡한 데이터베이스가 필요 없습니다.

---

## 2. 주요 작동 원리

템포는 다음과 같은 흐름으로 데이터를 처리합니다.

1.  **데이터 수집 (Data Ingestion):** 애플리케이션에 심어진 에이전트(예: OpenTelemetry Collector, Grafana Agent)가 요청 데이터를 수집합니다.

- 핵심 역할: 에이전트는 데이터를 무작정 보내기 전에 **배칭**(Batching)을 통해 네트워크 부하를 줄이고, 필요한 데이터만 골라내는 샘플링을 수행합니다.

2.  **Ingester (데이터 수용 및 가공):** 전송된 트래킹 데이터를 메모리에 일시적으로 저장하고 인덱싱합니다.
3.  **Backend Storage (압축 및 영구 저장):** Ingester에 쌓인 데이터가 일정 크기가 되면 하나의 Block으로 묶어 압축합니다. 이후 압축된 데이터를 S3와 같은 저렴한 저장소로 보냅니다.
4.  **Querier (데이터 조회 및 시각화):** 사용자가 Grafana UI에서 특정 ID로 검색하면, 저장소에서 데이터를 찾아 시각화해 줍니다.

## 3. 사용 시 장점

- **대규모 데이터 처리:** 인덱싱을 최소화하는 설계 덕분에 초당 수백만 개의 스팬(Span)을 처리할 수 있습니다.
  - span: 단일 서비스 내에서 수행된 하나의 작업 단위
  - 예를 들어 온라인 쇼핑몰에서 물건을 주문할 때 전체 주문 과정=Trace이고 사용자 인증-상품 재고 확인-카드 결제 요청-결제 정보 저장 이 4가지의 부분 과정이 span들이다.

> **Tempo**는 수집된 추적 데이터를 쌓아두는 **창고**입니다.
